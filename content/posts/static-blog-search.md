静态博客没有数据库，并不等于只能逐页翻找。只要构建时已经知道全部文章，就可以把标题、摘要、分类和标签组成一份很小的索引，在浏览器里完成搜索。

## 搜索的不是整篇正文

第一版索引只保留读者真正会用来判断文章的字段：

```ts
type SearchPost = {
  slug: string;
  title: string;
  summary: string;
  category: string;
  tags: string[];
};
```

搜索时把这些字段拼成一个字符串，再统一转成小写。相比把每篇正文全部发送到浏览器，这种方式体积更小，也能减少正文中的普通词造成的无关匹配。

## 分类负责方向，标签负责细节

分类应该少而稳定，例如前端、爬虫、AI 和随笔。标签则可以更具体，例如 Next.js、CSS、数据清洗和 JSON Schema。

分类太多时，读者面对的是另一种形式的搜索框；标签完全代替分类时，网站又会失去清楚的内容地图。两者分工以后，归档页可以先用四个方向帮助读者定位，再用关键词处理更具体的问题。

## 保留键盘与空结果路径

即时搜索不能只考虑鼠标点击。这个博客使用 `/` 聚焦搜索框，`Escape` 清空并退出。筛选结果通过 `aria-live` 报告数量，空结果则提供“显示全部文章”，不会让读者停在死胡同里。

还要注意三种边界：

1. 搜索框输入前，不要显示“没有结果”的加载闪烁。
2. 分类和关键词必须同时生效，重置时也要一起清空。
3. 手机没有键盘快捷键，需要保留足够大的搜索入口和点击目标。

## 什么时候需要真正的搜索服务

几十到几百篇文章时，浏览器内筛选通常已经足够。文章达到数千篇、需要正文分词、拼音匹配、模糊排序或跨语言检索时，再考虑构建倒排索引或接入专门的搜索服务。

先用最小方案验证读者如何查找内容，往往比提前维护一个搜索后端更划算。
