聊天窗口允许模型自由表达，程序接口却需要稳定字段。把大模型接进工作流时，真正的边界不是“能不能返回 JSON”，而是返回值能否被解析、校验、追踪，并在失败时安全退出。

## 先从消费方定义结构

不要先写一段很长的提示词，再猜程序会收到什么。先列出下游真正需要的字段、类型和约束。

```json
{
  "type": "object",
  "additionalProperties": false,
  "properties": {
    "title": { "type": "string", "minLength": 1 },
    "summary": { "type": "string", "minLength": 1 },
    "tags": {
      "type": "array",
      "items": { "type": "string" },
      "maxItems": 5
    }
  },
  "required": ["title", "summary", "tags"]
}
```

JSON Schema 可以同时描述数据结构和验证约束。模型、接口和业务代码共享同一份结构时，错误会更早暴露。

## 解析成功不等于数据有效

`JSON.parse()` 只能说明文本符合 JSON 语法，不能保证 `tags` 一定是数组，也不能保证缺失的标题可用。完整流程应该分为：

1. 检查请求是否成功，并记录请求标识。
2. 解析模型返回值。
3. 使用 schema 验证字段、类型和范围。
4. 执行业务规则，例如标题不能和已有内容重复。
5. 只有全部通过后才写入数据库或触发下一步。

把模型输出直接用于数据库写入、命令执行或对外发送，会把一次格式错误放大成真实副作用。

## 重试要带着失败原因

网络错误可以原样重试，结构校验失败则应把具体问题反馈给下一次请求，例如“缺少 summary”或“tags 不是字符串数组”。不要只说“格式不对”，也不要无限重试。

建议为整个任务设置次数和时间上限。超过上限后保留原始响应与校验错误，进入人工检查，而不是自动猜一个默认值继续执行。

## 为降级设计明确结果

不同场景需要不同的失败回退：

- 页面摘要生成失败，可以显示人工摘要或暂时隐藏摘要。
- 内容分类失败，可以进入“待分类”，但不能悄悄归入错误栏目。
- 自动发送消息失败，应停止并等待确认，不能生成另一段内容直接发送。

好的降级不是“什么都继续”，而是让可逆步骤继续，让有外部副作用的步骤停下来。

## 记录什么才能排查

日志至少需要模型或版本、schema 版本、请求耗时、重试次数、校验错误和最终状态。涉及隐私的数据应先脱敏，不要为了调试把完整用户输入永久保存。

当结构发生变化时，给 schema 增加版本号。这样历史数据和新程序之间的差异才有据可查。
